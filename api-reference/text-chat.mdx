# 文本对话 (Chat Completions)

```
POST https://llm.ziy.cc/v1/chat/completions
```

KUNPO API 完全兼容 OpenAI Chat Completions 格式，你可以直接使用 OpenAI 官方 SDK 或任何兼容客户端。

## 认证

所有请求必须在 `Authorization` 头中携带 API Key：

```
Authorization: Bearer sk-你的密钥
```

---

## 请求参数

### 通用参数

<ParamField path="model" type="string" required>
  模型 ID。海外模型使用 `供应商/模型名` 格式（如 `anthropic/claude-sonnet-4.6`），国内模型直接使用名称（如 `DeepSeek-R1`）。具体可用模型请在控制台模型市场查看。
</ParamField>

<ParamField path="messages" type="array" required>
  对话消息数组，每条消息包含 `role` 和 `content` 字段。

  支持的 `role` 值：`system`、`user`、`assistant`
</ParamField>

<ParamField path="stream" type="boolean" default="false">
  是否启用 SSE 流式输出。启用后响应以 `data: {...}` 格式逐 token 返回。
</ParamField>

<ParamField path="temperature" type="number" default="1.0">
  采样温度，范围 0-2。值越高输出越随机，值越低越确定。
</ParamField>

<ParamField path="max_tokens" type="integer">
  生成的最大 token 数。不同模型有不同上限。
</ParamField>

<ParamField path="top_p" type="number" default="1.0">
  核采样参数。与 `temperature` 二选一使用。
</ParamField>

### Gemini 专属参数

KUNPO API 支持通过 `extra_body` 透传 Google Gemini 专属参数：

<ParamField path="extra_body.generationConfig.thinkingConfig" type="object">
  思考模式配置。开启后模型会进行深度推理再回答。

  ```json
  "extra_body": {
    "generationConfig": {
      "thinkingConfig": {
        "thinkingBudget": 8000
      }
    }
  }
  ```

  <Accordion title="thinkingBudget 取值说明">
    | 值 | 含义 |
    |---|------|
    | `0` | 关闭思考模式（默认） |
    | `1` ~ `65536` | 思考 token 预算，越大推理越深入但响应越慢 |

    建议值：`8000`（平衡速度与质量）、`16000`（复杂推理任务）
  </Accordion>
</ParamField>

<Warning>
  `thinkingBudget` 仅对 Gemini 2.5 Pro / 2.5 Flash 系列有效。开启思考模式后，`temperature` 和 `top_p` 会被忽略（Google 强制设为 1.0）。
</Warning>

<ParamField path="extra_body.generationConfig.responseMimeType" type="string">
  控制输出格式。可选值：`application/json`（强制 JSON 输出）。
</ParamField>

<ParamField path="extra_body.safetySettings" type="array">
  安全过滤级别配置。每个对象包含 `category` 和 `threshold` 字段。

  <Accordion title="可配置的安全类别">
    | category | 说明 |
    |----------|------|
    | `HARM_CATEGORY_HARASSMENT` | 骚扰内容 |
    | `HARM_CATEGORY_HATE_SPEECH` | 仇恨言论 |
    | `HARM_CATEGORY_SEXUALLY_EXPLICIT` | 色情内容 |
    | `HARM_CATEGORY_DANGEROUS_CONTENT` | 危险内容 |
  </Accordion>

  <Accordion title="threshold 可选值">
    | 值 | 说明 |
    |----|------|
    | `BLOCK_NONE` | 不过滤 |
    | `BLOCK_LOW_AND_ABOVE` | 过滤低及以上风险 |
    | `BLOCK_MEDIUM_AND_ABOVE` | 过滤中及以上风险 |
    | `BLOCK_ONLY_HIGH` | 仅过滤高风险（默认） |
  </Accordion>
</ParamField>

## 请求示例

<CodeGroup>
  ```bash 基础对话
  curl -X POST https://llm.ziy.cc/v1/chat/completions \
    -H "Authorization: Bearer sk-你的密钥" \
    -H "Content-Type: application/json" \
    -d '{
      "model": "anthropic/claude-sonnet-4.6",
      "messages": [
        {"role": "system", "content": "你是一个有帮助的助手"},
        {"role": "user", "content": "什么是大语言模型？"}
      ]
    }'
  ```

  ```bash 多轮对话
  curl -X POST https://llm.ziy.cc/v1/chat/completions \
    -H "Authorization: Bearer sk-你的密钥" \
    -H "Content-Type: application/json" \
    -d '{
      "model": "anthropic/claude-sonnet-4.6",
      "messages": [
        {"role": "system", "content": "你是一个有帮助的助手"},
        {"role": "user", "content": "什么是大语言模型？"},
        {"role": "assistant", "content": "大语言模型（LLM）是一种基于深度学习的自然语言处理模型..."},
        {"role": "user", "content": "它和传统的NLP模型有什么区别？"}
      ]
    }'
  ```

  ```bash 流式输出
  curl -X POST https://llm.ziy.cc/v1/chat/completions \
    -H "Authorization: Bearer sk-你的密钥" \
    -H "Content-Type: application/json" \
    -d '{
      "model": "anthropic/claude-sonnet-4.6",
      "stream": true,
      "messages": [
        {"role": "user", "content": "用三句话介绍量子计算"}
      ]
    }'
  ```
</CodeGroup>

## 响应格式

### 非流式响应

```json
{
  "id": "chatcmpl-xxxxx",
  "object": "chat.completion",
  "created": 1711900000,
  "model": "anthropic/claude-sonnet-4.6",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "大语言模型（LLM）是一种基于深度学习的..."
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 24,
    "completion_tokens": 156,
    "total_tokens": 180
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 请求唯一标识 |
| `object` | string | 固定值 `chat.completion` |
| `created` | integer | 创建时间 Unix 时间戳 |
| `model` | string | 实际使用的模型名 |
| `choices` | array | 结果数组，通常只包含一个 |
| `choices[].message` | object | 助手回复消息，包含 `role` 和 `content` |
| `choices[].finish_reason` | string | 结束原因：`stop`（正常结束）、`length`（达到长度上限） |
| `usage` | object | Token 用量统计 |

### 流式响应

启用 `stream: true` 后，响应以 SSE（Server-Sent Events）格式返回：

```
data: {"id":"chatcmpl-xxxxx","choices":[{"delta":{"content":"大"},"index":0}]}

data: {"id":"chatcmpl-xxxxx","choices":[{"delta":{"content":"语言"},"index":0}]}

data: {"id":"chatcmpl-xxxxx","choices":[{"delta":{"content":"模型"},"index":0}]}

data: [DONE]
```

流式模式下每条 `data` 中的 `delta` 对象增量返回 `content` 片段，直至收到 `data: [DONE]` 标志结束。

## 视觉理解（多模态）

支持在消息中传入图片，用于视觉理解任务：

```bash
curl -X POST https://llm.ziy.cc/v1/chat/completions \
  -H "Authorization: Bearer sk-你的密钥" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "openai/gpt-5.4",
    "messages": [
      {
        "role": "user",
        "content": [
          {"type": "text", "text": "描述这张图片的内容"},
          {
            "type": "image_url",
            "image_url": {"url": "https://example.com/image.jpg"}
          }
        ]
      }
    ]
  }'
```

<Note>
  视觉理解能力取决于所选模型。支持视觉输入的模型包括 GPT-5.4 系列、Claude 系列、Gemini 系列、Qwen-VL 系列等。
</Note>

## DeepSeek 系列

KUNPO 平台上的 DeepSeek 模型提供两种协议格式：

- **不带 `c/` 前缀**（如 `DeepSeek-R1`）— 原生 OpenAI 兼容接口，通过 `/v1/chat/completions` 调用。
- **带 `c/` 前缀**（如 `c/deepseek-v4-pro`）— Claude 兼容接口，客户端使用 Anthropic 原生协议，但底层模型仍为 DeepSeek。

`c/` 前缀模型的详细用法（模型列表、价格、调用示例、选择建议）请参见 [Claude Messages API — DeepSeek 兼容模型](/api-reference/claude-messages#deepseek-兼容模型c-前缀)。

不带前缀的 DeepSeek 模型调用方式与其他模型一致：

```bash
curl -X POST https://llm.ziy.cc/v1/chat/completions \
  -H "Authorization: Bearer sk-你的密钥" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "DeepSeek-R1",
    "messages": [
      {"role": "user", "content": "帮我分析这段代码的时间复杂度：def merge_sort(arr): ..."}
    ]
  }'
```

## 错误处理

| HTTP 状态码 | 说明 | 处理方式 |
|:-----------:|------|----------|
| `400` | 请求参数错误 | 检查 JSON 格式和必填字段 |
| `401` | API Key 无效或过期 | 检查 Authorization 头 |
| `429` | 请求频率超限 | 降低请求频率，稍后重试 |
| `500` | 上游服务错误 | 稍后重试或更换模型 |
| `502` / `504` | 网关超时 | 大上下文请求可能需要较长时间，请耐心等待 |

错误响应格式遵循 OpenAI 标准：

```json
{
  "error": {
    "message": "Invalid API key provided",
    "type": "invalid_request_error",
    "code": "invalid_api_key"
  }
}
```
